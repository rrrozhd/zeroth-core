    ```markdown
    # Artifacts

    *Added in v4.0*

    The artifact store lets execution units and agents externalize large binary outputs (files, images, serialized models) so they travel outside the normal node output payload. Artifacts are stored on the filesystem and retrievable via the REST API.

    ## How It Works

    When a node produces a large output, it can call the artifact store to externalize the data. The store returns an `ArtifactKey` that travels through the graph as a lightweight reference. Downstream nodes or external clients retrieve the actual content by key.

    ## Key Components

    - **`FilesystemArtifactStore`** -- Default store implementation writing artifacts to a configurable directory. Constructed during `bootstrap_service()` and available as `bootstrap.artifact_store`.
    - **`ArtifactKey`** -- Opaque identifier returned on externalization; used for retrieval.

    ## REST API

    - `GET /v1/artifacts/{artifact_id}` -- Retrieve an artifact by key. Returns raw binary content. Requires `run:read` permission. Returns 404 if not found, 503 if artifact store is not configured.

    ## Configuration

    The artifact store is configured via `bootstrap_service()`. If no store is configured, the REST endpoint returns 503.

    See the [API Reference](../reference/http-api.md) for endpoint details and the source code under `zeroth.core.artifacts` for implementation.
    ```

    **docs/concepts/templates.md:**
    ```markdown
    # Templates

    *Added in v4.0*

    The template registry stores and versions prompt templates by name, enabling teams to manage prompt content separately from graph structure. Templates support Jinja2 variable interpolation in a sandboxed environment that prevents injection attacks.

    ## How It Works

    Templates are registered with a name and version. Agent nodes reference a template by name (and optionally version) via `template_ref` on `AgentNodeData`. At runtime, the orchestrator resolves the template, renders it with the node's input variables, and passes the rendered prompt to the LLM. Secret variables are automatically redacted in audit records.

    ## Key Components

    - **`TemplateRegistry`** -- In-memory registry storing templates by name and version. Supports create, get (by name or name+version), and list operations. Constructed during `bootstrap_service()`.
    - **`TemplateRenderer`** -- Renders a template string with variables using Jinja2's `SandboxedEnvironment`. Prevents code execution in template expressions.

    ## REST API

    - `GET /v1/templates` -- List all registered templates. Requires `run:read` permission.
    - `POST /v1/templates` -- Register a new template (name, version, content). Requires `admin` permission. Returns 409 if the name+version already exists.
    - `GET /v1/templates/{name}` -- Get a template by name (latest version). Optional `?version=N` query parameter for specific version. Requires `run:read` permission.
    - `DELETE /v1/templates/{name}/{version}` -- Remove a specific template version. Requires `admin` permission.

    ## Configuration

    The template registry is configured via `bootstrap_service()`. If no registry is configured, REST endpoints return 503.

    See the [API Reference](../reference/http-api.md) for endpoint details and the source code under `zeroth.core.templates` for implementation.
    ```

    ```markdown
    # Parallel Execution

    *Added in v4.0*

    Parallel fan-out/fan-in lets a single node spawn N parallel branches from its output, execute them concurrently, and merge results at a synchronization barrier with deterministic ordering.

    ## How It Works

    A node marked as a `FanOutSource` produces a list of items. The parallel executor splits this list into branches, creates isolated execution contexts for each branch, and runs them concurrently. A synchronization barrier collects results in branch-index order. Two execution modes are supported: **best-effort** (all branches run to completion, failures collected) and **fail-fast** (first failure cancels remaining branches).

    ## Key Components

    - **`split_fan_out()`** -- Splits a fan-out source node's output into parallel branches with isolated contexts.
    - **`BranchRunner`** -- Executes a single branch with its own visit counts, audit trail, and failure tracking.
    - **`merge_fan_in()`** -- Collects branch results at the synchronization barrier, ordered by branch index.
    - **`FanOutSource`** -- Configuration on a node marking it as a parallel source, with branch count and mode settings.

    ## Branch Isolation

    Each branch gets its own:
    - Visit counts (loop detection is per-branch)
    - Audit trail entries
    - Failure tracking
    - Cost attribution (BudgetEnforcer pre-reserves per branch)

    Policy, audit, and contract validation apply independently per branch.

    ## Known Limitations

    **SubgraphNode cannot be used inside parallel branches.** If a parallel fan-out source node has a SubgraphNode as a downstream node, the fan-out is rejected with a `FanOutValidationError`. This restriction exists because subgraph execution requires the full orchestrator `_drive()` loop, which is not available inside branch coroutines. Subgraph composition and parallel execution work independently but cannot be nested.

    See the [API Reference](../reference/http-api.md) for endpoint details and the source code under `zeroth.core.parallel` for implementation.
    ```

    **docs/concepts/subgraph.md:**
    ```markdown
    # Subgraph Composition

    *Added in v4.0*

    A graph can reference another published graph as a nested subgraph node. The orchestrator enters the subgraph as a scoped execution that inherits governance, optionally shares thread memory, and propagates approvals back to the parent.

    ## How It Works

    A `SubgraphNode` in a parent graph references a child graph by name (and optionally version). At runtime, the `SubgraphResolver` looks up the deployed child graph, namespaces its node IDs to prevent collisions, and merges the parent's governance as a ceiling the child can restrict but not relax. The `SubgraphExecutor` creates a child `Run` linked to the parent via `parent_run_id` and drives execution through the orchestrator's `_drive()` loop.

    ## Key Components

    - **`SubgraphNode`** / **`SubgraphNodeData`** -- Node type referencing a child graph with `graph_ref`, `version`, `thread_participation`, and `max_depth` settings.
    - **`SubgraphResolver`** -- Resolves graph references via `DeploymentService`, namespaces node IDs with `subgraph:{ref}:{depth}:` prefix, and merges parent governance.
    - **`SubgraphExecutor`** -- Creates child Run with `parent_run_id`, drives execution, and returns output to parent.

    ## Thread Participation

    - **`inherit`** (default) -- Child shares the parent's `thread_id`, so agents in the subgraph participate in the same thread memory.
    - **`isolated`** -- Child gets a new thread, fully isolated from the parent's memory.

    ## Approval Propagation

    If a `HumanApprovalNode` inside a subgraph pauses execution, the parent run transitions to `WAITING_APPROVAL` with metadata linking to the pending subgraph. When the approval is resolved, the resume cascades from parent to child, the child completes, and its output flows back to the parent.

    ## Depth and Cycle Protection

    - `max_depth` (configurable, default 3, max 10) prevents unbounded nesting.
    - Visited-ref tracking detects cycles where a subgraph references itself or an ancestor.

    ## Known Limitations

    SubgraphNode cannot be used inside parallel fan-out branches (see [Parallel Execution](parallel.md#known-limitations)).

    See the [API Reference](../reference/http-api.md) for endpoint details and the source code under `zeroth.core.subgraph` for implementation.
    ```

    **docs/concepts/context-window.md:**
    ```markdown
    # Context Window Management

    *Added in v4.0*

    Context window management tracks token usage per agent node and triggers automatic compaction when a configurable threshold is reached, preventing LLM context overflow without manual intervention.

    ## How It Works

    After each LLM invocation, the token count is updated via `litellm.token_counter`. When cumulative tokens exceed the configured threshold, a compaction strategy runs to reduce context size. The compacted result persists in thread memory, and the original messages can optionally be archived.

    ## Key Components

    - **`ContextWindowManager`** -- Orchestrates token tracking and triggers compaction. Configured per-agent-node via `ContextWindowSettings` on `AgentNodeData`.
    - **`TruncationStrategy`** -- Drops oldest messages to fit within the token budget.
    - **`ObservationMaskingStrategy`** -- Replaces observation/tool-output content with a placeholder, preserving message structure while reducing tokens.
    - **`SummarizationStrategy`** -- Uses an LLM call to summarize the conversation history into a condensed form.

    ## Configuration

    Context window settings are per-agent-node, configured on `AgentNodeData.context_window_settings`:
    - `enabled` -- Whether context window management is active for this node.
    - `max_tokens` -- Token threshold triggering compaction.
    - `strategy` -- Which compaction strategy to use (`truncation`, `observation_masking`, or `summarization`).
    - `archive_originals` -- Whether to keep original messages after compaction.

    The global `context_window_enabled` flag on the orchestrator controls whether the feature is active at the platform level.

    See the [API Reference](../reference/http-api.md) for endpoint details and the source code under `zeroth.core.context_window` for implementation.
    ```

    ```markdown
    # Resilient HTTP Client

    *Added in v4.0*

    The platform-provided async HTTP client gives agent tools and execution units a resilient way to make external HTTP calls with automatic retry, circuit breaking, and connection pooling.

    ## How It Works

    The `ResilientHttpClient` wraps `httpx.AsyncClient` with three resilience layers: configurable retry with exponential backoff and jitter, per-endpoint circuit breakers that open after repeated failures, and shared or per-tenant connection pools with configurable limits. External HTTP calls are gated by policy capabilities, logged in audit records, and subject to rate limiting.

    ## Key Components

    - **`ResilientHttpClient`** -- Main client class providing `get()`, `post()`, `put()`, `delete()`, and `request()` methods with built-in resilience.
    - **`RetryConfig`** -- Configures retry behavior: max attempts, backoff base/max, jitter, retryable status codes.
    - **`CircuitBreaker`** -- Per-endpoint circuit breaker with configurable failure threshold and reset timeout. States: closed (normal), open (failing fast), half-open (probing).
    - **`ConnectionPoolConfig`** -- Limits for max connections and max keepalive connections.

    ## Security

    - External HTTP calls are gated by the `network_read` / `network_write` capabilities in the policy system.
    - Auth headers and tokens can be resolved automatically from `SecretResolver`.
    - All external calls are logged in audit records with URL, status code, and timing.

    The HTTP client is constructed during `bootstrap_service()` and available as `bootstrap.http_client` on the orchestrator.

    See the [API Reference](../reference/http-api.md) for endpoint details and the source code under `zeroth.core.http_client` for implementation.
    ```
