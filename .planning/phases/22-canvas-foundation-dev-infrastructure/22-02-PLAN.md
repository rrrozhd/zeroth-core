<objective>
Create the Studio graph authoring REST API with CRUD endpoints for workflows and a node-types registry endpoint, mounted on the existing FastAPI app.

Purpose: The Studio frontend (Plans 03-04) needs a backend API to save/load workflow graphs and discover available node types. This API wraps the existing GraphRepository with Studio-specific request/response schemas that include visual metadata (node positions, viewport state) not present in the core graph model.

Output: FastAPI router at `/api/studio/v1/` with 6 endpoints: list/create/get/update/delete workflows + list node types. Pydantic schemas for all request/response models. Tests verifying CRUD operations.

    ```python
    from pydantic import BaseModel, Field
    from typing import Any

    class StudioPosition(BaseModel):
        x: float = 0.0
        y: float = 0.0

    class StudioViewport(BaseModel):
        x: float = 0.0
        y: float = 0.0
        zoom: float = 1.0

    class StudioNodeResponse(BaseModel):
        id: str
        type: str  # One of 8 frontend visual types
        position: StudioPosition
        data: dict[str, Any] = Field(default_factory=dict)

    class StudioEdgeResponse(BaseModel):
        id: str
        source: str
        target: str
        source_handle: str | None = None
        target_handle: str | None = None

    class CreateWorkflowRequest(BaseModel):
        name: str = Field(min_length=1, max_length=200)

    class UpdateWorkflowRequest(BaseModel):
        name: str | None = None
        nodes: list[StudioNodeResponse] | None = None
        edges: list[StudioEdgeResponse] | None = None
        viewport: StudioViewport | None = None

    class WorkflowSummaryResponse(BaseModel):
        id: str
        name: str
        version: int
        status: str
        updated_at: str

    class WorkflowDetailResponse(BaseModel):
        id: str
        name: str
        version: int
        status: str
        nodes: list[StudioNodeResponse]
        edges: list[StudioEdgeResponse]
        viewport: StudioViewport
        updated_at: str

    class PortDefinitionResponse(BaseModel):
        id: str
        type: str
        direction: str
        label: str

    class NodeTypeResponse(BaseModel):
        type: str
        label: str
        category: str
        ports: list[PortDefinitionResponse]
    ```

    **src/zeroth/service/studio_api.py**: Create the FastAPI router (per D-08, D-09):

    Router prefix: `/api/studio/v1`, tags: `["studio"]`

    Endpoints:
    1. `GET /workflows` -- List all workflows. Call `repo.list()`, map each Graph to WorkflowSummaryResponse. Return list.
    2. `POST /workflows` -- Create workflow. Accept `CreateWorkflowRequest`. Generate UUID for graph_id. Create a minimal `Graph(graph_id=uuid, name=body.name, nodes=[], edges=[], metadata={"studio": {"viewport": {"x": 0, "y": 0, "zoom": 1}, "node_positions": {}}})`. Call `repo.save(graph)`. Return 201 with WorkflowDetailResponse.
    3. `GET /workflows/{workflow_id}` -- Get workflow detail. Call `repo.get(workflow_id)`. If None, raise HTTPException(404). Map Graph to WorkflowDetailResponse. Node positions come from `graph.metadata.get("studio", {}).get("node_positions", {})`. Viewport from `graph.metadata.get("studio", {}).get("viewport", {})`.
    4. `PUT /workflows/{workflow_id}` -- Update workflow. Call `repo.get(workflow_id)`. If None, raise 404. Update name if provided. If nodes/edges/viewport provided, update graph.metadata["studio"] with positions and viewport data. For nodes, update both the graph nodes (creating lightweight placeholder nodes for visual-only types) and the position map. Call `repo.save(graph)`. Return WorkflowDetailResponse.
    5. `DELETE /workflows/{workflow_id}` -- Delete workflow. For now, archive it: call `repo.get()`, set status to ARCHIVED, call `repo.save()`. Return 204.
    6. `GET /node-types` -- Return the 8 node type definitions with port configs. This is a static registry, no database access needed. Return `list[NodeTypeResponse]`.

    The Studio API stores visual metadata (positions, viewport) in `graph.metadata["studio"]` to avoid modifying the core graph models. This is the mapping layer between frontend's 8 visual types and backend's 3 execution types (per anti-pattern from RESEARCH.md).

    For the node type registry, define all 8 types with their ports matching the frontend NODE_TYPE_REGISTRY:
    - `start`: category "flow", ports: [output-control]
    - `end`: category "flow", ports: [input-control]
    - `agent`: category "core", ports: [input-data, output-data]
    - `executionUnit`: category "core", ports: [input-data, output-data]
    - `approvalGate`: category "core", ports: [input-control, output-control]
    - `memoryResource`: category "data", ports: [input-data, output-data]
    - `conditionBranch`: category "flow", ports: [input-data, output-true, output-false]
    - `dataMapping`: category "data", ports: [input-data, output-data]

    Access the GraphRepository from `request.app.state.bootstrap.graph_repository`.
