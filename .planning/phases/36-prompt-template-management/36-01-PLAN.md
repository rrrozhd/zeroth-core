<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Template author -> Registry | Template strings are authored by graph builders (trusted users), but must still be validated for syntax and sandboxed for safety |
| Template variables -> Renderer | Variable values come from node input/run state/memory -- potentially containing user-supplied data that must not escape the sandbox |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-36-01 | Elevation of Privilege | TemplateRenderer | mitigate | Use `SandboxedEnvironment` which blocks `__class__`, `__bases__`, `__subclasses__` dunder attribute access at AST level. Catch `SecurityError` and wrap as `TemplateRenderError`. |
| T-36-02 | Information Disclosure | TemplateRenderer | mitigate | Use `StrictUndefined` so undefined variables raise errors rather than silently rendering empty (which could leak partial/incorrect data to LLM). |
| T-36-03 | Denial of Service | TemplateRenderer | accept | Jinja2 sandbox limits recursion. Templates are authored by trusted graph builders, not end users. No custom extensions allowed per D-04. Risk is low. |
| T-36-04 | Tampering | TemplateRegistry | mitigate | PromptTemplate model is `frozen=True` -- registered templates cannot be mutated after creation. Duplicate version registration raises error. |
</threat_model>
